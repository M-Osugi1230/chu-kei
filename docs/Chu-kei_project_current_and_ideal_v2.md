# 2. Chu-keiの目的

## 2.1 現在の定義

> 日本上場企業の中期経営計画を、戦略の違いから探し、比較・理解・調査する。

Chu-keiは中計PDFのリンク集ではない。ユーザーが調査課題から企業を発見し、戦略、数値目標、資本政策、進捗、過去計画を同じ画面で比較し、必ず原文へ戻れる状態を目指す。

## 2.2 投資助言サイトではない

Chu-keiは次を目的としない。

- 買い銘柄、売り銘柄の推奨
- 株価予測
- 割安、割高の判定
- 根拠のない総合ランキング
- 中計進捗率を企業価値や株価上昇確率として扱うこと

提供するのは、企業が公式に公表した情報を、出典、確認日、品質状態、レビュー判断とともに比較可能にすることである。

# 3. 想定ユーザー

## M&A・経営企画担当者

- M&A、事業ポートフォリオ、投資配分を横断比較する
- 同業他社の経営課題と戦略変更を確認する
- 提携候補、売却候補、業界プレーヤーの初期調査を行う

## 金融機関・FA・コンサルタント

- 顧客企業の競合、ベンチマーク企業を探す
- 経営課題別に企業を抽出する
- 提案資料や企業分析の初期調査を効率化する

## 事業会社の経営者・管理職

- 同業他社の成長投資、人材投資、資本政策を比較する
- 自社の中計策定や見直しの参考にする

## 個人の企業研究ユーザー

- 中計の要点を短時間で理解する
- 複数企業の戦略の違いを比較する
- 原文資料へ戻りながら検証する

# 4. 現在のデータ構成

## 4.1 本番品質1,500社

本番品質企業は、少なくとも次を満たす。

- 公式PDFまたは会社公式IRページを確認
- 企業名または証券コードの一致を確認
- 資料名、公表日、最終確認日を登録
- 経営課題、成長戦略、財務目標、資本政策、株主還元等を構造化
- PDFページ番号または具体的な公式Web見出しを一次証跡として登録
- 数値、単位、対象年度を区別
- 進捗を評価できない場合は `not_comparable` 等で明示
- 一次レビューを記録
- 独立再検証を記録
- 自動選定、自動承認、自動本番昇格を使用しない

## 4.2 Coverageβ400社

Coverageβは企業探索用であり、分析済み企業として扱わない。

- 上場企業としての基本情報は登録済み
- 公式中計・中長期戦略資料は未特定または要再調査
- 品質要件を満たすまでは本番品質へ昇格しない
- 中計を公表していない企業も含み得る

400社を一律に本番化するのではなく、公式資料の有無を確認し、「未公表」「資料特定済み」「対象外」「要再確認」を分離する。

## 4.3 進捗データ

公開用フロントエンドには353件の進捗レコードがある。

今後は件数だけでなく、次を重視する。

- 当初目標、修正目標、最新見込み、確定実績の区別
- 売上高、利益、利益率、ROIC、投資額、累計値ごとの評価ロジック
- 会社範囲、会計基準、指標定義変更の記録
- 比較不能な目標への架空の進捗率禁止
- 進捗根拠と実績根拠の双方の一次証跡

# 5. 品質基盤

本番品質1,500社への拡張時、次を含む15ワークフローが成功した。

- Chu-kei v43 Quality Gate
- Apply Structured Source of Truth
- Chu-kei Data Contract Gate
- Normalize Chu-kei Data Contract
- Chu-kei Quality Dashboard Gate
- Chu-kei Quality Debt Budget
- Chu-kei Source Audit
- Chu-kei Source Evidence Candidates
- Chu-kei Review Governance
- Chu-kei Entity Identity Audit
- Chu-kei Production Repair Queue
- Chu-kei Release and Deployment Gate
- Chu-kei Search and Filter Contract
- Chu-kei Browser E2E
- Chu-kei Accessibility and Performance

## 品質原則

1. 公式一次資料のみを採用する
2. 未確認情報を推測で補完しない
3. 未開示、未抽出、要確認、比較不能を分ける
4. 件数のために品質基準を緩めない
5. 本番昇格は個別レビューと独立再検証を必須とする
6. データ変更は修正台帳とレビュー判断で追跡する
7. 公開表示と正本データの意味を一致させる

# 6. 現在のユーザー体験

## 実装済み

- 会社名、証券コード、キーワード検索
- 市場、業種、品質、戦略テーマによる絞り込み
- 保存企業のみ表示
- 2〜4社比較
- URLによる検索条件、比較状態、企業詳細の共有
- 最近見た企業、類似企業、最近追加された企業
- モバイル対応
- 公式資料への遷移
- 品質状態、資料公表日、確認日の表示
- 過去中計、進捗変更イベント、商用受付の基盤

## 残るUX課題

- 保存情報が端末内中心で、クラウド同期や案件分けがない
- 比較画面で重要差分を自動強調できない
- 前中計からの変更点が主力表示になっていない
- 進捗データの企業・指標カバレッジが不足している
- 実利用ファネルを十分に把握できていない
- 公開サイトとGitHub正本の同期に手動要素が残る

# 7. 理想状態

Chu-keiの理想は、次を一つの場所で完結することである。

1. 調査テーマから企業を発見する
2. 現行中計の要点を理解する
3. 同業他社と同じ軸で比較する
4. 目標と実績の差を確認する
5. 前中計からの戦略変更を確認する
6. 原文資料へ戻って検証する
7. 調査候補、メモ、担当者、状態を保存する
8. 更新時に変更点を受け取る
9. チームで調査結果を共有する
10. Excel、PDF、APIで再利用する

# 9. 1500社到達後のロードマップ

## Phase A: 正本と公開の完全同期

- README、基準文書、品質ダッシュボード、公開サイト、リリース記録の件数を一致させる
- mainマージ後の公開反映を自動化する
- 公開後の件数、リンク、表示、検索、比較を監視する
- 旧件数や旧品質説明を検出するドキュメント監査を追加する

## Phase B: Coverageβ400社の再調査

- 公式IRページを再確認する
- 中期経営計画、中長期戦略、事業計画・成長可能性資料を探索する
- 未公表、資料特定済み、対象外、要再確認を分類する
- 品質条件を満たす企業だけを個別レビューへ送る
- 400社すべてを本番化すること自体を目標にしない

## Phase C: 進捗・実績接続の拡大

- 本番1,500社から、定量KPIと確定実績を接続できる企業を抽出する
- 指標定義、会社範囲、単位、年度、会計基準を照合する
- 比較不能を安全に表現する
- 目標変更イベントを記録する
- 企業詳細と比較画面に時系列を表示する

## Phase D: 過去中計比較

- 主要企業の前中計を収集する
- 前中計の最終結果を達成、未達、変更、撤回、判定不能で記録する
- 現中計への戦略変更を構造化する
- 投資配分、資本政策、収益目標、事業ポートフォリオの変化を表示する

## Phase E: 継続利用型ワークスペース

- 保存リストの案件分け
- 調査メモ、担当者、ステータス
- クラウド同期
- チーム共有
- 更新通知
- Excel・PDF出力

## Phase F: 収益化検証

1. スポット調査レポート
2. Pro
3. Team
4. API・データライセンス

品質状態や出典上の制約を、収益化のために隠さない。

# 13. 更新ルール

次の場合は、この文書とREADMEを同じ変更単位で更新する。

- 掲載企業数が変わった
- 公式資料確認・構造化済み社数が変わった
- 本番品質、Coverageβの構成が変わった
- 進捗・実績接続件数が大きく変わった
- 品質基準または品質負債の定義が変わった
- 新しい主要機能を公開した
- 公開同期方式を変更した
- 収益化方針を変更した
- Chu-keiの目的または禁止事項を変更した

過去版は削除せず、Git履歴と文書リンクで追跡可能にする。
