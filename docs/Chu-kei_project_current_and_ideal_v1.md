# 1. 結論

Chu-keiは現在、**570社を検索できる中期経営計画データベース**から、**企業を発見し、保存し、比較し、中計目標と実績まで調査できるワークスペース**へ進化している段階にある。

すでに次の基盤は存在する。

- 日本上場企業570社の探索
- データ品質の4段階管理
- 公式一次資料とページ証跡を重視したデータ運用
- 未確認情報を推測で補完しない設計
- 検索、保存、比較、URL共有
- 中計目標と実績の一部接続
- データ契約、品質監査、ブラウザE2E、アクセシビリティ、性能監査

一方、理想に対する最大の不足は、検索できる企業数ではなく、**深く比較できる企業数**である。

現在は570社を検索できるが、主要論点まで構造化されている企業は110社、本番品質は30社である。目標と実績が接続されている企業も16社にとどまる。

したがって、Chu-keiは単なる試作品ではないが、本格的な企業調査インフラ、継続利用されるワークスペース、収益化サービスとしては発展途上である。

# 2. Chu-keiの目的

## 2.1 現在の定義

Chu-keiの目的は、次のとおりである。

> 日本上場企業の中期経営計画を、戦略の違いから探し、比較・理解・調査する。

Chu-keiは中計PDFを集めるだけの資料置き場ではない。

ユーザーが短時間で次のような企業を見つけ、違いを理解できるようにする。

- M&Aを成長戦略に使っている企業
- ROIC、ROE、資本コストを重視している企業
- 株主還元に積極的な企業
- 海外展開を進めている企業
- 新規事業を育成している企業
- DX・AIを経営戦略へ組み込んでいる企業
- 人的資本へ重点投資している企業
- 構造改革や事業ポートフォリオ改革を進めている企業
- 中計目標と実績を比較できる企業

## 2.2 投資助言サイトではない

Chu-keiは、投資助言、銘柄推奨、株価予測を目的としない。

次のような表示や訴求は行わない。

- 買い銘柄、売り銘柄の推奨
- 儲かる企業、勝てる銘柄という表現
- 割安株ランキング
- 株価の将来予測
- 根拠のない総合企業ランキング
- 中計進捗率を企業価値または株価上昇確率として扱うこと

Chu-keiが提供するのは、企業が公式に公表した情報を、出典と品質状態を明示して比較可能にすることである。

## 2.3 理想の提供価値

理想的なChu-keiは、企業名を知っている人だけが使うサイトではない。

ユーザーが抱える調査課題から、該当する企業を発見できる状態を目指す。

例:

- M&Aを積極活用している製造業を探す
- ROIC目標があり、事業ポートフォリオ改革を進めている企業を探す
- DOEまたは累進配当を採用している企業を比較する
- 中計初年度から進捗が遅れている企業を確認する
- 前中計未達後に、現中計で戦略を変更した企業を調べる
- 同業他社の投資配分、資本政策、目標水準を横断比較する

# 3. 想定ユーザーと利用目的

## 3.1 主要ペルソナ

### M&A・経営企画担当者

- 積極的にM&Aへ取り組む企業を探す
- 同業他社の成長投資、事業ポートフォリオ、資本政策を比較する
- 提携候補、売却候補、業界プレーヤーを調査する

### 金融機関・FA・コンサルタント

- 顧客企業の競合やベンチマーク企業を探す
- 経営課題別に企業を抽出する
- 提案資料や企業分析の初期調査を効率化する

### 事業会社の経営者・管理職

- 同業他社が何を経営課題としているかを把握する
- 投資、人材、株主還元、資本効率の方針を比較する
- 自社の中計策定や見直しの参考にする

### 個人の企業研究ユーザー

- 中計の要点を理解する
- 企業同士の戦略の違いを比較する
- 原文資料へ戻りながら調査する

## 3.2 共通するユーザーの目的

- 企業を発見する
- 中計の要点をつかむ
- 複数企業を同じ軸で比較する
- 目標と実績の差を確認する
- 過去中計からの変化を調べる
- 原文資料へ戻って検証する
- 調査候補を保存し、後から続ける

# 4. 現在のデータ構成

## 4.1 main基準の現在値

2026-07-12時点のmain基準は、次のとおりである。

| データ階層 | 現在 | 意味 |
|---|---:|---|
| 掲載企業 | 570社 | 企業探索が可能 |
| 中計ソース確認済み | 200社 | 公式IR起点または中計資料を確認 |
| 主要論点構造化済み | 110社 | 中計内容を共通項目に整理 |
| 本番 | 30社 | 公式資料、主要論点、数値、証跡、人手レビューを確認 |
| 詳細抽出済みβ | 80社 | 比較可能だが本番昇格前 |
| 一次確認β | 90社 | 公式IR起点を確認、本文の詳細抽出前 |
| カバレッジβ | 370社 | 企業探索用、中計資料未特定 |
| 進捗DB登録 | 44社・149行 | 中計の定量目標を登録 |
| 実績接続 | 16社・54行 | 目標に対する実績を接続 |

現在、Issue #41 / PR #42で第4構造化拡張バッチが進行している。

目標は、品質基準を維持しながら、構造化済み企業を110社から最大115社へ増やすことである。

未達でも件数を埋めるために品質基準を緩和しない。

## 4.2 4段階の品質階層

### 本番

- 公式一次資料を確認
- 中計資料名、計画期間、公表日を確認
- 主要論点を構造化
- 定量KPIを登録
- ページまたは公式図版の証跡を登録
- 人手レビューを実施
- 本番昇格判断を記録

### 詳細抽出済みβ

- 中計本文の主要論点を構造化済み
- 比較に利用できる
- 一部のページ証跡、日付、実績接続、別確認者レビューなどが残る場合がある
- 自動的には本番昇格しない

### 一次確認β

- 会社公式IRページまたは中計資料の起点を確認
- 中計本文の詳細抽出前
- 詳細な比較項目は未抽出を明示

### カバレッジβ

- 上場企業としての基本情報を確認
- 企業探索には利用できる
- 中計資料は未特定
- 品質スコアの算定対象外
- 分析済みのように見せない

## 4.3 重要な解釈

「本番30社」は、サイトに30社しか掲載されていないという意味ではない。

570社を検索できるが、情報の深さと確認状態が異なる。

- 30社: 高品質な比較・調査が可能
- 80社: 主要論点を比較できるがレビュー途中
- 90社: 公式資料の入口まで確認
- 370社: 企業探索専用

すべてを同一品質として見せないことが、Chu-keiの重要な設計原則である。

# 5. 理想のデータ構成

## 5.1 中期目標

| データ階層 | 理想水準 |
|---|---:|
| 掲載企業 | 570社以上を維持 |
| 中計ソース確認済み | 570社 |
| 主要論点構造化済み | 300社 |
| 本番品質 | 100社 |
| 進捗目標登録 | 100社以上 |
| 実績接続 | 50〜100社 |
| 過去中計比較 | 主要100社で2計画以上 |
| 更新履歴 | 本番企業100% |

## 5.2 理想における考え方

570社すべてを急いで詳細抽出する必要はない。

重要なのは、次の3点である。

1. 各企業について、現在の確認状態が正しく表示される
2. 比較価値が高い企業から深度を上げる
3. 本番企業では品質負債を残さない

掲載社数、構造化社数、本番社数は別々に管理し、社数を増やすために品質基準を緩めない。

# 6. データ品質と信頼性

## 6.1 現在の品質基盤

現在のリポジトリには、次の品質管理がある。

- v43 strict品質監査
- Data Contract Gate
- JSON Schema
- 品質スコア
- 出典登録監査
- 公式リンク監査
- レビュー・修正台帳監査
- 企業同一性・表記揺れ監査
- 検索・複合フィルター契約
- 保存・比較・URL復元契約
- ChromiumデスクトップE2E
- 390pxモバイルE2E
- WCAG A / AAアクセシビリティ監査
- 性能予算
- Netlify公開物・リリースマニフェスト監査
- データバンドルのSHA-256整合性確認

## 6.2 現在の品質負債

現在の基準値は、次のとおりである。

- 品質負債: 89項目
- 詳細抽出済みβのページ証跡不足: 48社

品質負債の主な内容は、次のようなものである。

- 資料公表日の未確定
- ページ番号または図版番号の不足
- 定量KPIの根拠不足
- 目標と実績の未接続
- 最終確認日の陳腐化
- 別確認者レビュー未完了
- 当初目標、修正目標、最新見込みの区別不足

品質負債は必ずしも誤情報を意味しない。

しかし、本番昇格を判断するうえで残された確認作業であり、継続的に削減する。

## 6.3 理想の品質状態

本番企業について、表示する重要な数値または文章ごとに次を追跡できる状態を目指す。

- 公式資料URL
- 資料名
- 資料公表日
- 対象ページ
- 原文または図表の位置
- 最終確認日
- 抽出状態
- レビュー状態
- 変更履歴
- 変更前値と変更後値
- 修正理由

本番30社から100社へ増やす際は、品質負債を抱えたまま本番企業数を増やさない。

本番階層では、重大な品質負債ゼロを目標とする。

# 8. 中計目標と進捗管理

## 8.1 現状

現在、44社について149件の中計目標が登録されている。

そのうち、16社・54件に実績が接続されている。

表示上は、次を分離している。

- 目標値
- 目標年度
- 実績値
- 実績年度
- 単位
- 単純進捗率
- 目標の根拠
- 実績の根拠
- 最終確認日

進捗率は実績÷目標の単純比率であり、達成確率、将来予測、企業評価を意味しない。

## 8.2 現在の不足

- 実績接続企業数が少ない
- 年度ごとの時系列が不足している
- 当初目標と修正目標の区別が不十分
- 会社予想と確定実績の区別が不十分
- 事業セグメント別KPIが不足している
- 定性施策の進捗を扱えていない
- 中計の撤回、延長、変更を十分に表現できない
- KPI定義変更の履歴管理が不足している

## 8.3 理想

企業ごとに、次の流れを追える状態を目指す。

```text
中計公表
↓
当初目標
↓
年度ごとの会社計画
↓
年度実績
↓
目標修正
↓
最終結果
↓
次期中計への引継ぎ
```

そのうえで、根拠付きで次の状態を表示する。

- 達成
- 未達
- 前倒し達成
- 目標引き上げ
- 目標引き下げ
- 計画延長
- 指標変更
- 計画撤回
- 判定不能

売上高、利益、ROIC、利益率、投資額、累計値などでは評価方法が異なるため、指標別の進捗ロジックを持つ。

# 9. 過去中計比較

## 9.1 現状

現在は、現行または直近の中計を中心に構造化している。

過去中計の体系的比較は、主要機能としては未完成である。

## 9.2 理想

主要企業について、前中計と現中計を次の軸で比較する。

| 比較項目 | 前中計 | 現中計 |
|---|---|---|
| 基本戦略 | 何を重視したか | 何へ変更したか |
| 売上目標 | 数値 | 数値 |
| 利益目標 | 数値 | 数値 |
| ROIC・ROE | 数値 | 数値 |
| 投資額 | 数値 | 数値 |
| 株主還元 | 方針 | 方針 |
| 事業ポートフォリオ | 方針 | 方針 |
| 最終結果 | 達成・未達 | 進行中 |
| 未達理由 | 原文要約 | - |
| 次期計画への反映 | 何を修正したか | - |

これにより、目標が高い企業ではなく、次のような企業特性を調査できるようにする。

- 計画達成力がある
- 未達後に戦略を修正できる
- 毎回同じ目標を繰り返している
- 戦略が頻繁に変わる
- 資本政策が一貫している
- 投資配分が大きく変化している

# 10. 開発・運用体制

## 10.1 現在の正本分担

### 公開サイト

公開サイトは、次の最新プロダクト基準とする。

- UX
- 画面構成
- 説明文
- 探索導線
- 比較体験
- 将来の有料導線

### GitHub

GitHubは、次の正本とする。

- 企業データ
- 出典
- 品質状態
- 監査結果
- 変更履歴
- レビュー判断
- 昇格判断
- 再現可能な検証コード

データの意味を変える場合は、公開サイトだけを修正しない。

GitHubのSchema、監査、変更台帳を先に更新する。

## 10.2 現在の運用課題

- 公開サイトとGitHubの件数に時間差が発生する
- READMEや完了レポートに旧件数が残る場合がある
- 公開サイト側の変更履歴とGitHub側の変更台帳が完全同期していない
- リリース単位の同期確認に手動要素がある

## 10.3 理想のリリースフロー

```text
公式資料の取得
↓
データ抽出
↓
変更台帳作成
↓
人手レビュー
↓
品質ゲート
↓
mainへマージ
↓
公開サイトへ自動反映
↓
リリース記録作成
↓
公開後監視
```

理想状態では、GitHub main、公開サイト、README、品質ダッシュボード、リリースマニフェストの件数と意味が一致する。

# 11. 利用状況の把握

## 11.1 現状

現在は、外部解析SDKを導入せず、ブラウザE2Eにより次の主要導線を検証している。

- 検索
- 保存
- 比較
- 共有リンク復元
- モバイル操作
- アクセシビリティ

## 11.2 課題

機能が動作することは確認できるが、実際のユーザーがどのように利用しているかは十分に把握できない。

特に、次が分からない。

- どの戦略カテゴリが利用されているか
- 検索から企業詳細へ進んでいるか
- 保存した企業を再訪しているか
- 何社比較しているか
- どこで離脱しているか
- 公式資料まで開いているか

## 11.3 理想

プライバシーに配慮しながら、次の指標を把握する。

- 検索実行率
- 検索結果から詳細閲覧への遷移率
- 保存率
- 比較開始率
- 共有率
- 再訪率
- 保存企業の再閲覧率
- 公式資料への遷移率
- 0件検索の内容
- モバイルとPCの利用差

掲載社数ではなく、ユーザーが調査を完了できたかをUXの主要評価軸にする。

# 12. 収益化

## 12.1 現在の方針

収益化は、次の順序で検討する。

1. スポット調査レポート
2. Pro
3. Team
4. API・データライセンス

現時点では、課金、アカウント、契約管理は本格実装していない。

## 12.2 無料版の理想

- 570社以上の企業探索
- 基本検索
- 戦略カテゴリ
- 企業詳細
- 2〜4社比較
- 公式資料リンク
- 品質表示
- 一部の中計目標・実績

## 12.3 Proの理想

- 保存企業のクラウド同期
- 高度な複合検索
- 比較結果のExcel・PDF出力
- 過去中計比較
- 進捗アラート
- 変更履歴
- 企業ウォッチリスト
- 詳細な定量KPI
- 調査メモ

## 12.4 Teamの理想

- チーム共有ワークスペース
- 案件別企業リスト
- 担当者
- コメント
- 社内メモ
- 調査履歴
- 権限管理
- 共通テンプレート
- CSV・Excel一括出力

## 12.5 API・データライセンスの理想

- 企業基本情報
- 中計構造化データ
- KPI目標・実績
- 戦略テーマ
- 変更履歴
- 品質状態
- 出典情報

## 12.6 収益化の前提

課金機能を急ぐより、先に次を達成する。

- 本番品質100社
- 継続的に利用できる調査ワークスペース
- 過去中計比較
- 十分な目標・実績接続
- 更新監視と変更履歴
- 公開サイトと正本の完全同期

品質状態や出典上の制約を、収益化のために隠さない。

# 14. 優先順位

## 最優先: 品質を守りながら比較可能企業を増やす

1. 現在の品質負債89項目を継続削減する
2. 本番30社の不足する公表日・ページ証跡を解消する
3. 詳細抽出済みβのページ証跡不足を削減する
4. 一次確認βから、条件を満たす企業だけを詳細抽出済みβへ上げる
5. 詳細抽出済みβをレビューし、本番100社を目指す

## 次点: 目標と実績の接続を拡大する

1. 進捗登録企業を44社から100社へ増やす
2. 実績接続企業を16社から50社以上へ増やす
3. 当初目標、修正目標、最新見込み、実績を分離する
4. 指標別の進捗評価ロジックを整備する

## 次点: 過去中計比較を主力機能にする

1. 主要企業の前中計を取得する
2. 現中計との戦略差分を構造化する
3. 達成、未達、修正、撤回を記録する
4. 企業詳細と比較画面へ表示する

## 次点: 調査ワークスペースを継続利用型へ進化させる

1. 保存リストの案件分け
2. メモ、ステータス、担当者
3. クラウド同期
4. チーム共有
5. エクスポート

## 次点: 公開運用を一本化する

1. mainと公開サイトの自動同期
2. README、品質ダッシュボード、リリース記録の件数同期
3. 公開後のリンク監視と更新検知
4. 変更履歴の自動生成

## 最後に収益化を接続する

1. スポット調査レポート
2. Pro
3. Team
4. API・データライセンス

# 16. この文書の更新ルール

次の場合は、この文書を更新する。

- 掲載企業数が変わった
- 中計ソース確認済み社数が変わった
- 構造化済み社数が変わった
- 本番、詳細抽出済みβ、一次確認β、カバレッジβの構成が変わった
- 進捗目標または実績接続件数が大きく変わった
- 品質負債の基準値が変わった
- 新しい主要機能を公開した
- 収益化方針を変更した
- Chu-keiの目的または禁止事項を変更した

更新時は、冒頭の日付とバージョンを更新し、過去版を削除せずGit履歴で追跡可能にする。
